## 1 项目设计

### 1.1 模型设计

**架构选择：** 水文要素预报面向 **高维海表格点场**（多变量、长序列、强空间相关），需在有限算力下同时刻画 **大尺度环流** 与 **局地锋面/流轴**。本仓库主模型 **`HybridElementForecastModel`**（`src/element_forecasting/model.py`）以 **时空 ViT 式 Transformer** 为骨干：先在 **空间维** 将每个时刻的场嵌入为 Patch token，再在 **时间维** 对同一空间位置的 token 序列做自注意力编码；该路径记为 **`SpatioTemporalTransformerBranch`**。为进一步强化 **局地纹理** 与 **平流型形变**，在主干之外并行 **双专家混合（MoE）**：**UNet 专家**（通道维展平后的轻量 U-Net）与 **TrajGRU 专家**（可学习多链路流场 + `grid_sample` 重采样 + 门控递推）。专家输出经 **样本级 softmax 门控** 与 **残差系数 $\beta$** 与主干融合，并对 **流速分量（ssu/ssv）** 做通道聚焦增强。该设计与业务侧要求的 **多变量、多步输出** 一致，单段前向输出步长由 `model.yaml` 中 `output_steps` 控制（默认与 `input_steps` 对齐为 24），再通过推理器 **`predict_long_horizon`** 滚动扩展至 **72 小时** 等业务评估口径。

**多变量处理：** 输入/输出变量组由 `var_names` 统一配置（与 `configs/element_forecasting/model.yaml` 一致，README 中亦记作 `sst/sss/ssu/ssv`），与 NetCDF 变量名一致。在 **`SpatioTemporalTransformerBranch`** 末端，采用 **共享卷积头 `shared_head`** + **按变量独立的 `var_heads`**（每变量一套卷积），属于 **多任务学习（MTL）** 范式：共享主干学习 **海表多要素在时空上的联合表征**（温盐流之间的动力耦合由同一前向与联合损失共同约束），各变量头保留 **任务专属解码自由度**，便于在损失函数中对 **流速** 与 **温盐** 施加差异化权重（见 `train.yaml` 中 `var_loss_weights`、`moe_focus_vars`）。

**多输出长度设计：** 网络 **单段前向** 直接产生 **`output_steps` 步** 的张量 $\mathbf{P}\in\mathbb{R}^{B\times T_{out}\times C\times H\times W}$，避免逐步自回归链式乘积误差在训练阶段无界放大。评估关注的 **72 小时** 通过两类机制对齐：（1）**训练**：`rollout_steps=3` 将 72 拆为三段 24 小时滚动监督，配合 `rollout_gamma` 加大后段损失权重；（2）**推理**：`ElementForecastPredictor.predict_long_horizon` 以 24 步为块循环前向，并在块边界使用 **重叠线性融合** 抑制台阶效应。

### 1.2 数据处理设计

**数据清洗与预处理：** 原始 NetCDF 经项目 `data_preprocessing` 流水线完成哨兵值、缺测与一致性处理；要素预报训练/推理读取 **单一合并文件**，路径由 `data/processed/element_forecasting/path.txt` 给出。`ElementForecastWindowDataset`（`src/element_forecasting/dataset.py`）在 **时间维** 上以 **`input_steps` / `output_steps`** 构造滑窗样本，并对各变量按 **`element_forecasting_norm.json`** 做 **逐通道标准化**；同时利用有效掩膜构造 **`y_valid`**，供损失与评估在 **有效格点** 上统计。

**数据划分：** 默认 `split_mode=competition_years`，按 **预测目标时段所在自然年** 划分 train/val/test（`train.yaml` 中 `split_years`，默认 train 1994–2013、test 2014、val 2015），避免 **跨年窗口** 导致的 **时间泄漏**。`window_stride` 控制滑窗步长，在样本量与样本相关性之间折中。

### 1.3 优化设计

---

#### 1.3.1 空间 Patch 与海表多变量场编码

**引入动机：** 若在原始 $H\times W$ 像素网格上直接做全局自注意力，计算与显存随格点数平方增长，难以覆盖高分辨率海表场。将每个时刻的场 **分块（Patch）** 为少量空间 token，可在可控复杂度内保留 **大尺度环流结构**。

**原理说明：** 对每个时间片 $\mathbf{x}_t\in\mathbb{R}^{C\times H\times W}$，使用步长等于核大小的卷积

$$
\mathbf{Z}_t = \mathrm{Conv2d}(C\rightarrow d_{model},\;k=s,\;\mathrm{stride}=s)(\mathbf{x}_t)
$$

其中 $s=\texttt{spatial\_downsample}$（如 4）。等价于 **非重叠 Patch 嵌入**，空间 token 数 $N_{spa}=\lceil H/s\rceil\lceil W/s\rceil$。

**技术实现：** 见 `SpatioTemporalTransformerBranch.__init__` 中 `patch_embed`；空间位置使用可截断的 **`spa_pos_emb`**。多变量通道 $C$ 在嵌入前已按 `var_names` 堆叠，**同一嵌入层** 共享于所有变量，后续由 **`var_heads`** 分流解码。

**优势与局限性分析：** 显著降低显存并适配格点数据；**代价** 是固定步长下采样会 **压缩有效空间分辨率**，亚中尺度涡丝、极细锋面等信号在 Patch 表示中可能被 **欠采样**。`refine_head`、MoE 中的 UNet 分支与（可选）**多尺度辅分支**（见 **§1.3.3**）用于补偿，但 **不能** 在理论上保证「完全找回」原像素级细节。

---

#### 1.3.2 时间维建模与非因果历史编码

**原理说明（术语）：** 时间轴采用 **`nn.TransformerEncoder`**，在长度为 $T_{in}$ 的历史窗口内对时间 token 做 **全连接自注意力**（PyTorch 默认 **不施加** 下三角 **因果掩码**）。因此，应表述为 **非因果时间编码器（non-causal temporal encoder）** 或 **双向历史编码**：窗口内各时刻可相互依赖，**不同于** 语言模型解码器中「当前位置仅可见左侧 token」的 **严格因果注意力**。

**与「未来信息泄漏」的关系：** 本实现 **不存在将未来观测 $x_{t+k}$（$k>0$）喂入网络** 的路径；输入严格为 $\{x_{t-T_{in}+1},\ldots,x_t\}$，未来真值仅作为 **监督信号** 参与损失。**历史窗口内的双向混合** 并不引入对 $t$ 之后 **已观测场** 的访问；若审稿语境下的 *leakage* 特指「自回归训练中使用未来标签作为输入」，则本管线 **不满足** 该种泄漏定义。

**与自回归生成的差别：** 单段前向采用 **直接多步输出**（见 **§1.1**），而非逐步解码；滚动/长程在 **推理或 rollout 训练** 中另行组织（**§1.3.9**）。若需与 **因果自回归 LM** 严格对齐，可在时间层增加 **`attn_mask`（下三角）** 或改用 **因果卷积/状态空间层** —— 当前仓库以 **窗口式联合编码 + 多步监督** 为主，作为 **可插拔扩展** 另行论证。

**技术实现：** `temporal_encoder` 使用默认无掩码的 `TransformerEncoderLayer`；周期谐波与绝对时间编码（**§1.3.5**）为模型提供 **相位一致的时序先验**，与滑动窗口共同约束 **条件分布** $P(Y_{t+1:t+T_{out}}\mid X_{t-T_{in}+1:t})$ 的学习。

**优势与局限性分析：** 训练并行度高、实现简洁；局限性是 **历史内部注意力非因果**，若研究目标要求 **与逐步生成严格同构的训练动力学**，需显式引入因果掩码或分阶段策略并重新评估长程指标。

---

#### 1.3.3 时空分支融合与可选多尺度残差

**引入动机：** 单一 Patch 尺度可能对 **中尺度涡丝/锋面** 与 **大尺度背景场** 同时欠拟合。

**原理说明：** 当 `multi_scale_enabled=true` 时，增加 **辅分支** `patch_embed_aux`（更小 `aux_spatial_downsample`，如 8），在嵌入空间 **双线性对齐** 至主分支分辨率后，按 `multi_scale_fusion`（如 `residual_add`）与权重 `multi_scale_aux_weight` 融合。

**技术实现：** `SpatioTemporalTransformerBranch.forward` 中分支融合逻辑；默认配置中 **常关闭** 辅分支以抑制 **插值格纹**。

**优势与局限性分析：** 增强多尺度表达；局限性为计算量与显存上升，需调权避免辅分支噪声压制主分支。

---

#### 1.3.4 空间分块（Patching）参数与张量变换

**引入动机：** 明确张量形状与超参，便于复现与调参。

**原理说明：** 设批次大小 $B$，输入时间长度 $T_{in}$，变量数 $C$，空间 $H\times W$，Patch 步长 $s$。则每个时间片经嵌入后得到 $\frac{H}{s}\times\frac{W}{s}$ 个空间 token（实现中假设整除或由下采样卷积自然对齐），总 token 展平为 $N_{spa}$。张量形状演化为：

$$
(B,T_{in},C,H,W)\rightarrow (B\cdot T_{in},\,d_{model},\,H',W')\rightarrow (B\cdot T_{in},\,N_{spa},\,d_{model})
$$

**技术实现：** `model.py` 中 `forward` 对 `spa_tokens` 的 reshape 与 `spatial_encoder`；**注意**：本仓库 **未** 采用时间维 `unfold` 式 Patch（即 **不是** 将 $L$ 维序列切成 $P$ 个长度为 $K$ 的时间块），时间维组织为 **整段 $T_{in}$ 历史** 进入时间编码器。

**优势与局限性分析：** 空间 Patch 与海洋格点数据自然对齐；与「纯时间序列 Patch」文献需区分表述，避免评审误解。

---

#### 1.3.5 块嵌入、周期谐波时间编码与解码头

**引入动机：** 海表要素受 **日周期、周周期** 等影响显著，仅用整数时间索引不足以表达 **相位信息**。

**原理说明：** 对绝对时间索引构造多周期、多谐波特征：

$$
\phi_{p,k}(t)=\sin\!\left(\frac{2\pi k t}{P_p}\right),\quad \psi_{p,k}(t)=\cos\!\left(\frac{2\pi k t}{P_p}\right)
$$

其中 $P_p\in$ `periodic_periods`，$k=1\ldots$ `periodic_harmonics`。经线性层 `tod_emb` 映射到 $d_{model}$，与 `tem_pos_emb` 一并注入时间 token。

**技术实现：** `_build_periodic_time_features` + `tod_emb`；解码侧 **`shared_head`** + **`var_heads`** 输出各变量多步场；可选 **`DepthwiseSeparableRefineHead`** 做输出空间细化。

**优势与局限性分析：** 显式注入 **物理可解释** 的周期先验；局限性为周期列表需与数据采样间隔（小时）一致配置。

---

#### 1.3.6 空间—时间 Transformer 信息交互

**引入动机：** 需在 **空间依赖**（同一时刻场内的远程相关）与 **时间依赖**（同一空间点上的演变）之间分配模型容量。

**原理说明：** **前半数** `TransformerEncoderLayer` 作用于 **空间 token 序列**（固定时刻内 $N_{spa}$ 个位置）；**后半数** 作用于 **时间 token 序列**（固定空间位置上 $T_{in}$ 步）。该 **分层策略** 在 `num_layers` 固定时控制两类注意力的深度。

**技术实现：** `spatial_encoder` 与 `temporal_encoder` 串联；时间编码输出取 **最后一步** `tail`，再 reshape 回 $H\times W$（必要时双线性上采样至原分辨率）。

**优势与局限性分析：** 结构清晰、易调试；局限性为时空交互顺序为 **先空后时**，与「时空联合注意力」类模型不同，需在文档中如实陈述。

---

#### 1.3.7 双专家系统（UNet + TrajGRU）

**引入动机：** 单一 Transformer 对 **极细尺度纹理** 与 **强平流输送** 的同步刻画存在折中；引入 **结构先验不同** 的两条专家路径，由数据驱动学习 **何时信哪一路**。

**专家 A — UNetExpert：** 将 $(T_{in}\cdot C)$ 通道拼成单帧二维输入，经 U 型编码—解码与跳连，输出 $(T_{out},C,H,W)$，强化 **空间多尺度局部对比度**。

**专家 B — TrajGRUExpert：** 以 `TrajGRUCell` 沿时间递推隐状态，流场分支输出多链路位移，`grid_sample` 对隐状态重采样后门控更新，强化 **流致形变**。

**技术实现：** `UNetExpert`、`TrajGRUExpert`、`HybridElementForecastModel` 中 `moe_enabled` 分支；checkpoint 中专家分支键名 **`pred_convlstm`** 为 **历史兼容命名**，实际为 TrajGRU 输出。

**预测空间对齐与隐空间异构：** UNet 与 TrajGRU 的中间特征维度、感受野与归纳偏置（**纹理/多尺度卷积** vs **流致形变与递推**）并不相同，本实现 **未** 在隐层增加独立的「跨专家投影对齐」模块；两路均在各自 **`head`** 中解码为与主干 **同形** 的物理预报张量 $\mathbb{R}^{B\times T_{out}\times C\times H\times W}$，随后在 **预测空间** 由门控加权并与主干融合（见 **§1.3.8**）。**优点** 是端到端损失与辅助损失直接约束 **量纲一致、可比较的多步场**；**代价** 是若两支路在优化上竞争，需依赖 $\beta$、辅助损失权重、AMP 与梯度裁剪维持稳定。**可演进方向** 包括专家瓶颈层对齐、对比正则或分阶段预训练。

**优势与局限性分析：** 互补时空归纳偏置；局限性为参数量与训练稳定性需靠 **辅助损失权重**（`loss_aux_unet_weight`、`loss_aux_convlstm_weight`）与 **梯度裁剪/AMP** 等联合约束。

#### 1.3.8 门控融合与 HybridElementForecastModel 集成流程

**引入动机：** 不同海况下 **局地纹理结构** 与 **平流主导输送** 的相对重要性不同，需要 **随样本变化** 的专家权重；门控网络将这一分配表述为 **对两类动力学机制的软路由**。

**原理说明：** 取 **输入序列最后一帧** $\mathbf{x}_{t}$ 的 **全局空间平均** $\bar{\mathbf{x}}\in\mathbb{R}^{B\times C}$，作为 **当前海表状态** 的粗粒度摘要（含温盐流多通道信息）。经 `gate_mlp`（代码：`Linear(C, max(16,2C))` → **GELU** → `Linear(max(16,2C), 2)`）得到 logits，**softmax** 得 $(w_{unet},w_{traj})$。**物理解释性叙述（假设性，非单独消融结论）：** 在 **强流、强梯度或锋面活跃** 的情形下，数据驱动可能抬高 TrajGRU 权重（强调 **流致位移/形变**）；在 **中尺度纹理与对比度** 更突出的情形下，可能更依赖 UNet。**数值上** 门控完全由损失与数据共同学习，答辩时应与 **门控权重分布的可视化统计** 一并呈现则更稳妥。

专家混合 $\mathbf{P}_{exp}=w_{unet}\mathbf{P}_{unet}+w_{traj}\mathbf{P}_{traj}$，再与主干做 **残差式注入**：

$$
\mathbf{P}=\mathbf{P}_{transformer}+\beta\,\mathbf{P}_{exp}
$$

其中 $\beta=\texttt{moe\_residual\_beta}$，可理解为 **专家混合项相对 Transformer 主干的注入强度**（类 **门限/温度** 的全局系数）：过小则退化为近似纯主干，过大则压制共享时空表征。对 `moe_focus_channel_indices` 指定通道（通常为 **ssu/ssv**）再叠加 **聚焦增强项**，使矢量流场在专家残差上获得 **额外修正自由度**。

**技术实现：** `HybridElementForecastModel.forward`；返回字典含 `pred`、`pred_transformer`、`pred_unet`、`pred_convlstm`、`moe_gate_weights`。配置项 `moe_transformer_fusion_alpha` **保留** 于构造函数，**当前 forward 以 $\beta$ 残差融合为主**，对外说明与代码需一致。

**优势与局限性分析：** 单前向内完成 **自适应专家混合**；局限性为门控仅依赖 **末帧全局池化**，对 **空间非均匀海况** 的细粒度路由能力有限，且 $\beta$ 与辅助权重仍属 **需验证集标定** 的超参数。

#### 1.3.9 多步预报、滚动训练与长程重叠推断

**引入动机：** 业务与评估主指标常面向 **72 小时** 等长 horizon；若仅训练单段 24 小时，**长程误差累积** 与 **曝光偏差** 在部署时放大。

**原理说明：**

1. **直接多步输出**：单段网络输出 $T_{out}=24$，避免逐步单步链式乘积。  
2. **Rollout 训练**：`rollout_steps=3`，三段拼接覆盖 72 小时；`rollout_gamma>1` 加大后段损失。  
3. **Scheduled Sampling**：以概率 $\epsilon$ 用真值作为下一段输入，$\epsilon$ 随 epoch 余弦衰减，缓解训练—推理分布偏移。  
4. **长程推理**：`predict_long_horizon` 循环调用 `predict`，步长为 `model_steps - overlap_steps`（代码内变量 `stride`），重叠区按 $\alpha\in[0,1]$ 线性融合。

**技术实现：** `trainer.py` 中 rollout 与采样逻辑；`predictor.py` 中 `predict_long_horizon`；Web `element.py` 中 `target_steps=72`、`overlap_steps=4`。

**联合建模表述：** 多变量未来场由 **同一前向、共享时空主干与联合损失** 同步预测（见 **§1.1**）；**温盐流之间的动力耦合** 由共享时空表征、`var_heads`、差异化损失权重及（可选）流速通道专家聚焦 **显式保留在联合优化目标中**，与多任务学习叙述一致。

---

#### 1.3.10 训练稳定性与工程优化技术

**动机：** 大模型、高分辨率与海表长序列易导致 **显存峰值** 与 **训练不稳定**；在数据驱动主干之外，**物理与结构先验损失** 与架构设计 **并列**，用于约束预报场的 **平滑性、散度行为、梯度与边缘一致性** 等，使优化目标与海洋动力 **粗粒度一致**。

**要点（与 `train.yaml` / `trainer.py` 对齐）：**

| 技术 | 作用 |
|------|------|
| **AMP 混合精度** | 降低显存、加速矩阵运算 |
| **梯度累积 `grad_accum_steps`** | 在有限显存下模拟更大有效 batch |
| **多进程 DataLoader `num_workers`** | 提高 CPU 侧解码与预处理吞吐 |
| **验证集 `val_target_steps=72`** | 与业务评估 horizon 对齐 |
| **重叠融合训练开关** | `overlap_blend_enabled` 与推理一致，减轻块边界伪影 |
| **物理与结构损失组合** | 散度、平滑、梯度、边缘、可微平流等（`physics_losses.py`），直接加在 **多步预报张量** 上，与 NRMSE 等数据项加权求和 |

**局限性分析：** **RevIN**、**激活检查点**、**Cosine Restarts** 等若未在当前 `trainer` 默认启用，则 **不在此宣称已实现**；若后续接入，应同步更新本段与配置说明。物理损失系数需在 **物理可信度与数值稳定性** 之间标定，过大可能损害拟合主指标。

### 1.4 实验设置、数据集与基准评估（严格响应提交要求）

#### 1.4.1 实验数据集规模、要素类型与场景分布

要素预报使用 **单一合并 NetCDF**，变量与评估任务定义一致（以 `var_names` 为准）；时间划分见 **§1.2**。标准化文件为 `data/processed/normalization/element_forecasting_norm.json`。

#### 1.4.2 实验评估指标体系

主指标 **NRMSE (%)**（掩膜有效区）；训练过程记录 **分变量 NRMSE、Bias**，并支持 **前 24h / 后 24h** 分段指标，防止总分掩盖后段漂移。损失项见 `train.yaml` 与 `physics_losses.py`。

## 2 项目前后端系统介绍

### 2.1 前后端框架介绍

本系统 **要素预报 Web 链路** 采用 **前后端分离**：后端 **FastAPI + Uvicorn**，前端 **Vue 3 + Vite**，开发态由 **`scripts/07_web_run.py`** 启动（可先轮询 `/healthz` 再启前端，见脚本说明）。

#### 2.1.1 后端

**框架：** Python **FastAPI**（非 Flask），应用入口 `src/web/backend/app/main.py`，要素路由 `src/web/backend/app/routers/element.py`，前缀 **`/api`**。

**核心逻辑：** `POST /api/predict` 使用 **`ElementForecastPredictor.predict_long_horizon`** 生成 **72 步** 预测，将 **`pred` 真值、掩膜、变量名** 写入进程内 **`state.prediction_cache`**；`GET /api/predict/{session_id}/step/{step_idx}` 按步返回切片，避免单次 JSON 体积过大。

#### 2.1.2 前端

**技术栈：** **Vue 3** 组件 + **Plotly.js** 渲染空间场与曲线（如 `ForecastPanel.vue`、`useForecast.ts`）。

**（图4 要素预报前端界面示意图 — 排版时插入）**

**交互流程：** 页面加载请求 **`/api/default-data-path`** 填充默认数据路径 → 用户校验数据集 → **`POST /api/predict`** 触发推理 → 时间滑条驱动 **`GET .../step/...`** 刷新热力图 → **`GET .../curve`** 更新区域均值曲线。
